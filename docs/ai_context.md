# Zeip MVP – Technical Summary (ADR Lite)

## 1. Architecture & Stack

- **Frontend**: Next.js 16 (App Router, `app/`), TypeScript, Tailwind CSS.
- **Backend**: Supabase (Postgres, Auth, Realtime).
- **Hosting**: Vercel; environment variables (в **Settings → Environment Variables** для Production / Preview):
  - `NEXT_PUBLIC_SUPABASE_URL` — URL проекта Supabase (например `https://xxxx.supabase.co`).
  - `NEXT_PUBLIC_SUPABASE_ANON_KEY` — anon public key из Supabase → Project Settings → API.
- **Client data access**: `src/lib/supabaseClient.ts` (browser-safe anon client).

### Деплой на Vercel (чеклист)

1. **Репозиторий**: корень Git должен совпадать с папкой Next-приложения **или** в Vercel указать **Root Directory** = `my-app` (если в монорепо выше лежит только оболочка).
2. **Framework Preset**: Next.js; команда сборки по умолчанию `next build`, выход `.next`.
3. **Переменные**: без обеих `NEXT_PUBLIC_*` приложение на проде падает при обращении к Supabase (пустой URL / ключ) — проверить, что они заданы для **Production** и сделан **Redeploy** после добавления.
4. **Локально перед пушем**: `cd my-app && npm run build` — та же команда, что на Vercel; если билд падает локально, на Vercel упадёт так же.
5. **Подключение домена**: Vercel → Project → Domains; в Supabase → Authentication → URL Configuration добавить production URL в **Site URL** и **Redirect URLs**, иначе OAuth / magic link дадут ошибку редиректа.
6. **CLI** (альтернатива Git): из `my-app` после `npx vercel login` и `npx vercel link` — `npx vercel --prod` (переменные задаются в дашборде или через `vercel env`).

## 2. Core Domain & Data Model (Supabase)

### Auth & Profiles
- `auth.users`: базовая таблица пользователей (Supabase).
- `public.profiles`:
  - `id uuid` – primary key, используется как ID профиля на фронте и в ссылках `/profiles/[id]`.
  - `auth_user_id uuid unique` – связь с `auth.users.id`.
  - Основные поля: `full_name`, `country`, `city`, `avatar_url (пока нет)`.
  - Профессиональные поля:
    - `industry` – отрасль (справочник подгружается из `industry_catalog`; ручной вариант хранится через `industry_other`).
    - `industry_other` – ручной ввод, если выбрано `industry = 'Другое'` (при сохранении добавляется в `industry_catalog`).
    - `subindustry` – подотрасль, зависит от выбранной отрасли (при выборе «Другое» добавляется в `subindustry_catalog`).
    - `role_title` – **профессия** (UI-слой; физически поле то же самое).
    - `current_status` – текущий статус пользователя (значения в UI: `Учащийся`, `Сотрудник в компании`, `Предприниматель`, `Фрилансер`).
    - `experience_years` – стаж (число лет).
    - `skills` – поле “О себе” (текст).
    - `looking_for`, `can_help_with`, `interested_in` – legacy-текстовые поля в БД (в форме `/profile` скрыты, но пока не удалены из схемы).
  - Presence:
    - `last_seen_at timestamptz` – последняя активность пользователя для статуса Онлайн/Оффлайн.
  - **Рейтинг (баллы)**:
    - используем `profiles.rating_count` как **целочисленные баллы** (“Рейтинг X” в UI);
    - начисление реализовано на бэке через триггеры (см. `docs/rating.sql`):
      - +1 за каждое заполненное поле профиля (пересчёт при сохранении профиля, применяется дельта),
      - +1 за указанное гео (один раз, когда впервые появляется активная локация),
      - +1 за отправку сообщения (1 раз в день по Мск; считается и `posts`, и `messages`),
      - +1 за ответ на входящее сообщение (1 раз в день по Мск; только `messages`, если предыдущее сообщение в чате было от другого отправителя).

### Специализации
> **Удалено:** специализации удалены из UI и из БД (таблицы `specialties/user_specialties` и колонка `profiles.domain`).

### Геолокации
- `public.locations`:
  - `id uuid` – маркер.
  - `user_id uuid` → `profiles.id`.
  - `lat`, `lng` – точные координаты, сохранённые пользователем.
  - `city text`, `is_active boolean`.
- Использование:
  - для карты специалистов на главной;
  - редактируется в `/profile` через `ProfileLocationPicker` (react-leaflet).
  - приватность на главной карте:
    - реальный `lat/lng` не показывается напрямую;
    - пин отображается в детерминированно-случайной позиции в радиусе ~250 м от точки (`PartnerMap`, obfuscation per user).

### Посты / Общий чат
- `public.posts`:
  - `id uuid`.
  - `author_id uuid` → `profiles.id`.
  - `title text` (сейчас "Общий чат").
  - `body text` – сообщение (ограничиваем на UI до 1000 символов).
  - `city text` (сейчас не отображаем).
  - `specialty_id uuid` (поле в БД сохранено, но в текущем UI **не используется**).
  - `created_at timestamptz`.
- Отображение/поведение:
  - Лента слева на главной:
    - грузим 20 последних постов.
    - карточка показывает:
      - круг‑аватар с **первой буквой имени** (клик открывает pop‑up профиля, как и имя),
      - имя + **индикатор Онлайн/Оффлайн**,
      - бейдж профессии (`profiles.role_title`); при наличии суммарного стажа по этой профессии (из `profiles.experience_years` + `profile_work`) в бейдж добавляется текст вида `· N лет`,
      - текст сообщения (200 символов + “Читать далее”).
    - время отправки под текстом: `ДД.ММ.ГГГГ, ЧЧ:ММ` (таймзона по городу текущего пользователя; fallback — таймзона браузера).
  - Фильтры/поиск:
    - поп‑ап “Поиск по специалистам” (в шапке колонки карты) фильтрует **и ленту, и карту** по:
      - профессии (`profiles.role_title`),
      - отрасли/подотрасли (`profiles.industry/subindustry`),
      - текущему статусу (`profiles.current_status`),
    - хранение в `localStorage`: `feed_filters`.
  - Отправка:
    - авторизованные пользователи могут отправлять сообщения.
    - отправка по кнопке и по `Enter` (Shift+Enter = перенос).
    - новые сообщения добавляются в начало списка, UI хранит последние 20.

### Личные чаты / Мессенджер
- `public.chats`:
  - `id uuid`.
  - `is_group boolean` – сейчас используем только `false` (личные диалоги 1:1).
  - `title text` (для групповых чатов на будущее).
  - `created_by uuid` → `profiles.id`.
- `public.chat_members`:
  - `chat_id uuid` → `chats.id`.
  - `user_id uuid` → `profiles.id`.
  - `role text` (member/admin/owner; сейчас используется минимально).
- `public.messages`:
  - `id uuid`.
  - `chat_id uuid` → `chats.id`.
  - `sender_id uuid` → `profiles.id`.
  - `content text` – до 1000 символов (ограничение на UI).
  - `created_at timestamptz`.
  - `is_read boolean default false` – пока логика частичная (см. ниже).

#### Поведение мессенджера
- Личный чат создаётся лениво:
  - при открытии диалога (через правый столбец / карту / профиль) ищем чат `is_group=false` с двумя участниками (текущий пользователь + собеседник);
  - если не найден — создаём `chats` + две записи в `chat_members`.
- Загрузка сообщений:
  - при открытии чата загружаем последние 5 сообщений (ASC для корректного порядка).
  - UI показывает облака:
    - сообщения текущего пользователя (`sender_id === currentUser.profileId`) — справа, **emerald‑градиент** (фон `#10B981` / градиент в стиле UI);
    - сообщения собеседника — слева, светло‑серые.
    - время отправки: `ДД.ММ.ГГГГ, ЧЧ:ММ` (таймзона определяется по городу пользователя; fallback на таймзону браузера).
- Отправка:
  - `INSERT` в `messages` с `sender_id = currentUser.profileId`.
  - локально добавляем сообщение в `chatMessages`.
  - автоскролл к последнему сообщению.
- Realtime:
  - включён Realtime для таблицы `messages`.
  - подписка `postgres_changes` на `INSERT`:
    - игнорируем сообщения, где `sender_id === currentUser.profileId` (уже есть локально).
    - проверяем, что `currentUser` — участник чата через `chat_members`.
    - если чат сейчас открыт (`activeChatId`) → добавляем сообщение в `chatMessages` + автоскролл.
    - иначе инкрементируем счётчик непрочитанных для собеседника (`unreadByUser[userId] += 1`).
- “Мои чаты” (3‑й столбец):
  - показываются **только** собеседники текущего пользователя:
    - находим все `chat_members` для `user_id = currentProfileId`,
    - по этим `chat_id` вытаскиваем других участников и их `profiles`,
    - формируем список `chatList` с `chatId + profile + lastMessageAt`.
  - порядок:
    - сортируем сверху вниз по `lastMessageAt` (последнее сообщение → выше),
    - при отправке/получении нового сообщения соответствующий чат поднимается наверх.
  - каждая плашка (карточка):
    - аватар с инициалом, индикатор Онлайн/Оффлайн у края аватара, при непрочитанных — **красный** числовой бейдж на аватаре,
    - имя (клик — pop‑up профиля), при `rating_count > 0` — звезда и число справа в строке имени,
    - профессия и город строками ниже,
    - при `unread > 0` карточка подсвечивается (emerald‑фон/бордер); счётчик непрочитанных также отражается в **мобильной** нижней навигации на вкладке «Мои чаты».
  - кликом по плашке открываем чат; при этом `unreadByUser[profileId]` сбрасывается в 0.
  - по клику на **имя** в плашке открывается pop‑up профиля (без открытия чата).
- Окно чата:
  - фиксированная высота, скролл сообщений внутри.
  - отображается поверх карты/селекторов (`z-index` выше выпадающих меню карты).
  - закрывается:
    - по крестику,
    - по клику вне окна,
    - по клавише `Esc`.
  - по клику на имя в шапке открывается pop‑up профиля (как в ленте/списке чатов).
  - в шапке чата рядом с именем показывается Онлайн/Оффлайн индикатор.

## 3. Главная страница (`src/app/page.tsx`)

- Три основных блока (на **desktop**, `lg:` и выше): flex, на всю высоту контента под хедером, фон страницы `gray-100`, боковые панели **по ~320px (`lg:w-80`)** с тенью и вертикальным бордером к центру:
  1. **Левая колонка**: общий чат / лента постов (шапка с иконкой, форма внизу с `border-t` и фоном `gray-50`).
  2. **Центр**: карта; у контейнера карты **`id="main-map"`** (якорь для скролла/навигации при необходимости).
  3. **Правая колонка**: «Мои чаты» или заголовок «Контакты» в режиме `?contacts=1` + всплывающее окно личного чата.
- **Мобильная вёрстка** (`< lg`): одновременно видна **одна** колонка по выбранной вкладке; внизу фиксированная навигация `MainMobileNav` (`src/components/MainMobileNav.tsx`): **Чат**, **Карта**, **Контакты**, **Мои чаты** (с бейджем суммарных непрочитанных). У основного контента отступ снизу под навбар (`pb-*`). Переключение вкладки «Контакты» делает `router.push('/?contacts=1')`; переключение на другие вкладки при активном режиме контактов вызывает **сброс query** (тот же путь, что и у кнопки «Сбросить», без принудительной смены вкладки — см. ниже).
- Синхронизация `?contacts=1`: как и раньше, читается из `window.location.search` + подписка на `popstate` и кастомное событие **`locationchange`** (патч `history.pushState` / `replaceState`), чтобы сработали и `next/router`, и прямые вызовы History API.
- При появлении режима контактов в URL вкладка мобильного UI выставляется в **«Контакты»** (`useEffect` от `contactsOnlyMode`).
- Кнопка **«Сбросить»** в правой колонке (только в режиме контактов): снимает `contacts` из URL и дополнительно переключает мобильную вкладку на **«Мои чаты»**.
- Окно личного чата на широких экранах позиционируется с учётом ширины правой колонки (`lg:right-[336px]` и т.п.).
- Данные загружаются через Supabase в одном `useEffect` (posts, profiles, текущий профиль).
- Дополнительно:
  - механизм автоподписки на Realtime для `messages`.
  - автоскролл в окне чата при новых сообщениях.
  - карточка профиля из общего чата:
    - по клику на имя в общем чате открывается плавающий поп‑ап справа от сообщения (фиксированная позиция по `getBoundingClientRect`), с кнопками “Профиль” и “Написать”.
  - pop‑up профиля также открывается:
    - по клику на имя в списке “Мои чаты”,
    - по клику на имя в шапке открытого окна личного чата.
- Фильтры/поиск специалистов:
  - поп‑ап фильтров (иконка настроек в шапке колонки карты) фильтрует ленту + карту, хранится в `localStorage` (`feed_filters`).
  - в фильтрах есть поле “Текущий статус”.
- Селектор города:
  - в **`TopBar`** (центр шапки), контекст `SelectedCityProvider` в `layout.tsx`;
  - “Россия” показывает общий вид страны;
  - значение хранится в `localStorage` (`selected_city`).

## 4. Профиль пользователя

### Страница редактирования `/profile`
- Клиентская страница, использует:
  - `supabase.auth.getUser()` для определения текущего пользователя.
  - поиск/создание `profiles` по `auth_user_id`.
  - загрузку/сохранение `locations` для координат.
- Поля формы:
  - Имя (лимит 40 символов), страна, город.
  - Текущий статус (dropdown): `Учащийся`, `Сотрудник в компании`, `Предприниматель`, `Фрилансер` (А-Я).
  - Профессия (кастомный dropdown; берётся из `public.profession_catalog`, сохраняется в `profiles.role_title`).
  - Другое по профессии: при выборе “Другое” появляется ручной ввод (лимит 40 символов); при сохранении новая профессия вставляется в `profession_catalog`.
  - Отрасль (кастомный dropdown; берётся из `industry_catalog`, сохраняется в `profiles.industry`).
  - Другое по отрасли: ручной ввод (через `industry_other`, лимит 40 символов); при сохранении новая отрасль вставляется в `industry_catalog`.
  - Подотрасль (кастомный dropdown; берётся из `subindustry_catalog`, сохраняется в `profiles.subindustry`).
  - Другое по подотрасли: появляется ручной ввод (лимит 40 символов); при сохранении новая подотрасль вставляется в `subindustry_catalog`.
  - Поле “О себе” (`skills`) с лимитом 600 символов и счётчиком `X/600`.
  - Поля “Что хочешь найти на платформе”, “Чем можешь помочь другим”, “Какие специалисты...” удалены из UI страницы `/profile`.
  - Локация на карте (react-leaflet):
    - клик по карте задаёт `lat/lng`;
    - выбранный пин можно перетаскивать (draggable marker).
- Повторяемая Группа 2:
  - можно добавить несколько наборов “Профессия/Отрасль/Подотрасль/Стаж” (до 5 блоков суммарно, считая первый базовый блок).
  - дополнительные блоки сохраняются в `public.profile_work`.
- При сохранении:
  - обновляем `profiles` с учётом отрасли/подотрасли и “Другое”;
  - остаёмся на странице `/profile` (без редиректа на `/`);
  - кнопка:
    - после успешного сохранения становится зелёной “Профиль сохранен”;
    - при новых изменениях снова становится синей “Сохранить профиль”;
  - создаём/обновляем запись в `locations` (по `profiles.id`).

### Публичная страница `/profiles/[id]`
- Только чтение, без редактирования.
- Показывает:
  - имя, город/страну,
  - отрасль (с учётом `industry_other` как кастомной подписи),
  - подотрасль, **профессию**, стаж,
  - “Что умеет”, “Что ищет”, “Чем может помочь”, “Кто интересует”.
- В шапке:
  - большие кнопки:
    - **“Назад”** → `/`,
    - **“Добавить в контакты / Удалить”** (для чужого профиля, связка в `profile_contacts`),
    - **“Написать”** → `/ ?chat=PROFILE_ID` (главная автоматически откроет чат).

## 5. Карта специалистов (`PartnerMap`)

- Компонент `PartnerMap` рендерится только на клиенте (`dynamic(..., { ssr: false })`) из‑за Leaflet.
- Источник данных:
  - `locations` (координаты и `user_id`),
  - массив `profiles` из родителя; связываем по `profiles.id === locations.user_id`.
- Отображение маркеров:
  - используются **`Marker` + `L.divIcon`**, не `CircleMarker`;
  - визуал как в макете Figma: круг **48×48**, белая обводка **4px**, тень, внутри **первая буква имени** (кириллица через `toLocaleUpperCase('ru-RU')`), под кругом «ножка» 4×12px;
  - **заливка круга и ножки у всех пинов одна**: `#10B981`;
  - стили ховера (лёгкое увеличение круга) — в `globals.css` (классы `partner-map-*`, прозрачный фон у `leaflet-div-icon`).
- Кэш `L.DivIcon` по ключу **буква** (цвет фиксированный), чтобы не пересоздавать иконки без нужды.
- Tooltip:
  - при наведении показываем имя + индикатор Онлайн/Оффлайн и должность (offset скорректирован под высоту пина).
- Popup по клику:
  - имя + индикатор Онлайн/Оффлайн, отрасль, подотрасль, должность,
  - рейтинг: “Рейтинг X” (из `profiles.rating_count`),
  - отрасль/подотрасль/должность берём из `profiles.*` (первый блок Группы 2),
  - кнопка **“Профиль”** → `/profiles/{user_id}`,
  - кнопка **“Написать”** → открывает личный чат через `onOpenChat(user_id)`.
  - кнопка **“Добавить в контакты / Удалить”**.
  - поп‑ап открывается с первого клика, поведение закрытия стандартное для Leaflet.

## 6. Навигация и хедер (`TopBar`)

- `TopBar` рендерится в `layout.tsx` на всех страницах (липкий хедер, `z-index` выше карты/оверлеев при необходимости):
  - слева: логотип **`/zeip-logo.svg`** + текст **Zeip** (ссылка на `/`).
  - по центру: **селектор города** (`TopBarCitySelect`, тот же список городов и `localStorage` `selected_city`, что раньше был над картой).
  - справа: кнопка **«Контакты»** (иконка людей) — переход на `/?contacts=1`; для авторизованного пользователя — **бейдж** с числом записей в `profile_contacts` (запрос `select` с `count: 'exact', head: true` по `owner_id`; после добавления/удаления контакта счётчик обновляется через событие `zeip-profile-contacts-changed`).
  - если пользователь не залогинен → справа кнопка **«Войти»** → `/auth` (стиль emerald‑градиент).
  - если залогинен → **аватар с инициалом** + с `md:` имя (первое слово `full_name`); по клику — выпадающее меню:
    - «Профиль» → `/profile`,
    - «Контакты» → `/?contacts=1`,
    - «Выйти» → `supabase.auth.signOut()` + редирект на `/auth`.
  - меню закрывается по клику вне и по `Esc`.
- Состояние авторизации:
  - начальная загрузка `supabase.auth.getUser()` + профиль (`id`, `full_name` для UI и контактов).
  - подписка `supabase.auth.onAuthStateChange`, чтобы меню сразу отражало логин/логаут (при выходе счётчик контактов сбрасывается).
  - heartbeat presence: для авторизованного пользователя периодически обновляется `profiles.last_seen_at` (~каждые 30 сек + при фокусе/видимости вкладки).

## 7. Особые решения и ограничения

- **Логика данных не менялась** при обновлении UI: те же таблицы, RLS, Realtime на `messages`, ленивая модель чатов, обфускация координат на главной карте, маскирование мата в **общем чате** (личные сообщения без маски — по-прежнему правилу продукта).
- **SSR**: по-прежнему избегаем `useSearchParams` на сервере для `?chat=` / `?contacts=` — чтение через `window` и `useEffect`.
- **Стили**: основной фон приложения в оболочке `layout` — `bg-gray-100`; акцентные кнопки на главной и в хедере в духе макета Figma Make (emerald).
- **Leaflet / `window`**:
  - компоненты с картой и `location picker` рендерятся только на клиенте через `dynamic(..., { ssr: false })`.
  - query `?chat=` читается через `window.location.search` в `useEffect` на главной.
- **Realtime**:
  - включён для таблицы `messages`; канал создаётся на клиенте (главная страница).
- **Справочники (UI)**:
  - `public.profession_catalog` — профессии (главная и `/profile`).
  - `industry_catalog` / `subindustry_catalog` — отрасли и подотрасли.
  - стратегия **DB-first**: сначала БД, при успехе — обновление `localStorage`; при ошибке — кэш из `localStorage`; без кэша — fallback-списки в коде.
- **Ручные значения “Другое”**: при сохранении профиля новые значения попадают в `profession_catalog`, `industry_catalog`, `subindustry_catalog`.
- **Группа 2 / `profile_work`**: дополнительные блоки «работа» в `/profile`; публичное отображение всех блоков — отдельное улучшение.
- **Безопасность**: только `NEXT_PUBLIC_SUPABASE_ANON_KEY` на клиенте; service-role не используется.
- **RLS по чатам**: нужны политики select (`posts_select_all`, `messages_select_if_member`, `chat_members_select_if_member`, `chats_select_if_member`) — см. `docs/chat_select_policies.sql`.

## 8. Контакты (избранное)

- **БД**: `public.profile_contacts (owner_id, contact_profile_id, created_at)` — список контактов пользователя.
  - RLS: читать/добавлять/удалять может только владелец (по `profiles.auth_user_id = auth.uid()`).
  - SQL для создания: `docs/contacts.sql`.
- **UI**:
  - кнопка **“Добавить в контакты / Удалить”**:
    - в pop‑up профиля из общего чата,
    - в pop‑up маркера на карте,
    - на публичной странице профиля `/profiles/[id]` (для чужих профилей).
  - в меню пользователя (`TopBar`) пункт **“Контакты”** → `/?contacts=1`;
  - дополнительно в хедере отдельная кнопка с иконкой и **бейджем** количества контактов (см. §6).
- **Поведение**:
  - в режиме `?contacts=1`:
    - на карте остаются только пины профилей из контактов текущего пользователя;
    - в списке “Мои чаты” остаются только чаты с пользователями из контактов;
    - в правой колонке заголовок **«Контакты»** и кнопка **«Сбросить»**: убирает query `contacts`, возвращает полный список чатов и все пины; на мобильной вёрстке переключает вкладку на **«Мои чаты»**;
    - на мобильной нижней навигации режим контактов соответствует вкладке **«Контакты»** (синхронизация с URL).
